Lesson 04

Streams & Pipeline

Process files of any size with constant memory. The pattern that separates production code from tutorial code.

Why Streams Exist

You already know from Lesson 3 that readFile loads the entire file into memory. That's fine for a 50KB config file. But what about:

  • A 2GB server log you need to analyze?
  • A user uploading a 500MB video?
  • An API response streaming a CSV export with 10 million rows?

readFile on a 2GB file allocates 2GB of RAM. Your server handles 10 concurrent requests like this — congratulations, you need 20GB of RAM (and you'll hit the 2GB libuv limit anyway and crash). Streams solve this by processing data chunk by chunk — only a small piece lives in memory at any time.

ApproachMemory for 2GB fileWorks?
readFile ~2GB ❌ Crashes (libuv 2GB limit)
Stream pipeline ~64KB – 1MB (configurable) ✅ Constant memory

The Mental Model: Water Pipes

Think of streams like plumbing. Data flows from a source (faucet), through transforms (filters), into a destination (drain): 

// The stream pipeline pattern:
//
//   [Readable]  →  [Transform]  →  [Transform]  →  [Writable]
//    (source)       (filter)       (modify)       (destination)
//
//  Example: Read log file → filter errors → format CSV → write report

Node.js has four stream types:

TypeRoleExamples
Readable Source of data fs.createReadStream(), http.IncomingMessage (request body)
Writable Destination for data fs.createWriteStream(), http.ServerResponse
Transform Modify data in transit zlib.createGzip(), custom parsers/filters
Duplex Both read and write TCP sockets (net.Socket)

As a full-stack dev, you'll mostly use Readable and Writable streams created by Node APIs, and occasionally write custom Transform streams. The stream plumbing connects them.

pipeline() — The Right Way to Connect Streams

The pipeline function from node:stream/promises is your go-to. It:

  • Connects a chain of streams (readable → transforms → writable)
  • Handles backpressure automatically (pauses fast readers when writers are slow)
  • Handles errors — destroys all streams if any one fails (no leaked file descriptors)
  • Returns a Promise — works with async/await
import { createReadStream, createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';

// Copy a file of any size using constant memory (~64KB)
await pipeline(
  createReadStream('huge-input.log'),
  createWriteStream('huge-output.log')
);
Never use .pipe() in production. The older readable.pipe(writable) method does NOT destroy streams on error — you'll leak file descriptors. Always use pipeline() from node:stream/promises.

Backpressure: Why Streams Don't Blow Up Memory

Backpressure is the mechanism that keeps a fast reader from overwhelming a slow writer. Imagine reading from a fast SSD but writing to a slow network connection:

  • The writable stream has an internal buffer (controlled by highWaterMark)
  • When the buffer fills up, the readable stream pauses
  • When the buffer drains, the readable stream resumes
  • Memory stays bounded no matter how fast the source or how slow the destination
// The highWaterMark controls the buffer threshold
// File read streams default to 64KB (not the generic 16KB)

createReadStream('data.bin', { highWaterMark: 64 * 1024 })  // 64KB (default)
createReadStream('data.bin', { highWaterMark: 1024 * 1024 }) // 1MB (faster, more RAM)

pipeline() manages all of this for you. You don't need to manually handle 'drain' events or pause/resume logic.

Real-World Pattern 1: Compress and Serve a File

A common Express/Node server pattern — stream a large file to the client with gzip compression: 

import { createReadStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';
import { createGzip } from 'node:zlib';
import http from 'node:http';

const server = http.createServer(async (req, res) => {
  res.setHeader('Content-Encoding', 'gzip');
  res.setHeader('Content-Type', 'text/plain');

  try {
    await pipeline(
      createReadStream('server.log'),  // Readable: file
      createGzip(),                      // Transform: compress
      res                                 // Writable: HTTP response
    );
  } catch (err) {
    if (!res.headersSent) {
      res.statusCode = 500;
      res.end('Internal Server Error');
    }
  }
});

server.listen(3000);

This streams a file of any size to the client, compresses it on the fly, and uses constant memory. The HTTP response object (res) is itself a Writable stream — so pipeline connects directly to it.

Real-World Pattern 2: Copy a File

The simplest possible stream pipeline — a file copy:

import { createReadStream, createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';

async function copyFile(src, dest) {
  await pipeline(
    createReadStream(src),
    createWriteStream(dest)
  );
}

await copyFile('backup.sql', 'backup-copy.sql');
Note: For simple file copies, copyFile from node:fs/promises is simpler and uses OS-level optimizations. Use streams when you need to transform the data in transit (compress, parse, filter, encrypt).

Real-World Pattern 3: Process a Large Log File

This is the pattern you'll use in full-stack apps — process a large file line by line using constant memory. We'll use the modern for await...of syntax with readline: 

import { createReadStream } from 'node:fs';
import { createInterface } from 'node:readline';

const rl = createInterface({
  input: createReadStream('server.log'),
  crlfDelay: Infinity, // treat \r\n as a single line break
});

let errorCount = 0;

for await (const line of rl) {
  const entry = JSON.parse(line);
  if (entry.level === 'error') {
    errorCount++;
  }
}

console.log(`Found ${errorCount} errors`);

for await...of on a readline interface gives you one line at a time — no matter how large the file. Memory stays flat.

Real-World Pattern 4: Custom Transform Stream

When you need to modify data as it flows through a pipeline, you write a Transform stream. Here's one that filters JSON log lines to only pass through errors: 

import { Transform } from 'node:stream';
import { createReadStream, createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';

// Custom Transform: parse JSON lines and filter errors
const errorFilter = new Transform({
  transform(chunk, encoding, callback) {
    const line = chunk.toString();
    try {
      const entry = JSON.parse(line);
      if (entry.level === 'error') {
        // push() sends data downstream
        this.push(line + '\n');
      }
      // If we don't push, the chunk is dropped (filtered out)
    } catch {
      // Skip malformed lines
    }
    callback(); // Signal: ready for next chunk
  }
});

await pipeline(
  createReadStream('server.log'),
  errorFilter,
  createWriteStream('errors-only.log')
);

The Transform pattern:

  • this.push(data) — sends data downstream (call multiple times or zero times)
  • callback() — signals you're ready for the next chunk (must always be called)
  • callback(error) — signals an error and stops the pipeline

When to Use Streams vs readFile

ScenarioUseWhy
Small files (< 50MB), known size readFile Simpler code, file fits comfortably in memory
Large files (100MB+) Streams Constant memory regardless of file size
Unknown file size (user uploads) Streams Can't predict memory needed
Need to transform data in transit Streams Process while reading, no intermediate copies
Serving files over HTTP Streams Start sending before file is fully read (lower TTFB)
JSON config, small text files readFile Need the whole file to parse anyway
Rule of thumb: If the file could be larger than ~100MB, or its size is unknown (user-provided content), default to streams. The code is slightly more complex but it won't crash under load.

Important Gotchas

1. Streams live on node:fs, not node:fs/promises

createReadStream and createWriteStream are on the callback-based node:fs import. But you combine them with pipeline from node:stream/promises — which gives you the async/await interface: 

// This is the correct import pattern:
import { createReadStream, createWriteStream } from 'node:fs';         // streams
import { pipeline } from 'node:stream/promises';  // async pipeline

2. Stream Chunks Don't Respect Boundaries

A chunk from createReadStream is just a block of bytes — it can split mid-line, mid-word, or mid-UTF8 character. If you need line-by-line processing, use readline.createInterface() or write a custom line-splitter Transform.

3. Error Handling Is Automatic With pipeline()

A single try/catch around await pipeline(...) catches errors from any stream in the chain. If the readable fails (file not found), or the writable fails (disk full), or a transform throws — all streams are destroyed and the error propagates to your catch block. 

try {
  await pipeline(
    createReadStream('input.log'),
    createGzip(),
    createWriteStream('output.log.gz')
  );
} catch (err) {
  if (err.code === 'ENOENT') {
    console.error('Input file not found');
  } else if (err.code === 'ENOSPC') {
    console.error('Disk full');
  } else {
    throw err;
  }
}

4. highWaterMark Tuning

Stream TypeDefault highWaterMarkWhen to Change
File read stream 64KB Increase for large sequential reads (1MB+)
Generic readable 16KB Rarely needs changing
Object mode stream 16 objects Lower to apply backpressure sooner

Hands-On Exercise

Create a script called compress-file.js that takes a filename argument and creates a gzipped copy: 

// compress-file.js
// Usage: node compress-file.js server.log
// Output: server.log.gz

import { createReadStream, createWriteStream } from 'node:fs';
import { pipeline } from 'node:stream/promises';
import { createGzip } from 'node:zlib';

const input = process.argv[2];

if (!input) {
  console.error('Usage: node compress-file.js <filename>');
  process.exit(1);
}

try {
  await pipeline(
    createReadStream(input),
    createGzip(),
    createWriteStream(`${input}.gz`)
  );
  console.log(`✓ Compressed: ${input} → ${input}.gz`);
} catch (err) {
  console.error(`✗ Failed: ${err.message}`);
  process.exit(1);
}

Try it:

# Create a test file
node -e "import('node:fs').then(fs => fs.writeFileSync('test.txt', 'hello '.repeat(100000)))"
node compress-file.js test.txt
ls -la test.txt test.txt.gz   # Compare sizes
