Lesson 01

The fs Module — Three APIs, One Decision

Understanding why node:fs/promises is your default, and when the others earn their place.

The Mental Model

Node.js gives you one file system module (node:fs) that exposes the same operations through three different API surfaces. As a frontend dev, you've already internalized a similar pattern: fetch returns a Promise, XMLHttpRequest uses callbacks, and neither blocks the main thread. The fs module works the same way — except it also ships a synchronous API that blocks the event loop entirely.

API Surface Import Style Blocks Event Loop?
Promise (modern default) node:fs/promises async/await No
Callback (legacy) node:fs Error-first callbacks No
Synchronous node:fs (*Sync methods) Returns directly Yes

The node: Prefix

Always import with the node: prefix. It explicitly marks the import as a Node.js built-in and cannot be shadowed by an npm package of the same name. This is a security and clarity improvement — an attacker can't publish a package named fs and have your code accidentally pull it in. 

// ✓ Modern — explicit, secure
import { readFile } from 'node:fs/promises';

// ✗ Legacy — works but ambiguous
import { readFile } from 'fs/promises';

API 1: node:fs/promises — Your Default

The promise-based API became stable in Node 14. It integrates cleanly with async/await and top-level await (available in ES modules since Node 14.8). This is what you use for all server-side code. 

import { readFile, writeFile, mkdir } from 'node:fs/promises';

// Top-level await — no wrapper needed in ESM
const config = JSON.parse(
  await readFile('config.json', 'utf8')
);

await mkdir('output/reports', { recursive: true });

await writeFile(
  'output/reports/summary.json',
  JSON.stringify(config, null, 2),
  'utf8'
);
Why not the callback API? The callback API predates Promises and leads to deeply nested error-first callbacks. It still works and you'll see it in older code, but for new code it offers no advantage over fs/promises. The Node.js docs themselves recommend the promise API for new projects.

API 2: Callback API — Legacy, Still Async

The callback API is the original fs surface. It doesn't block the event loop (good), but it produces the "callback pyramid" you've seen in pre-Promise Node code. You'll encounter it in older codebases and libraries. 

import fs from 'node:fs';

// Error-first callback pattern
fs.readFile('config.json', 'utf8', (err, data) => {
  if (err) {
    console.error('Failed:', err.code);
    return;
  }
  console.log(JSON.parse(data));
});

Your app.js currently uses this pattern with fs.mkdir. It works, but the promise version is cleaner and composable: 

// Your current code (callback)
fs.mkdir("src", (err) => {
  if (err) return console.log(err);
  console.log("done");
});

// Modern equivalent (promise)
import { mkdir } from 'node:fs/promises';

try {
  await mkdir('src', { recursive: true });
  console.log('done');
} catch (err) {
  console.error(err.code);
}

API 3: Synchronous — CLI Scripts Only

Every async method has a *Sync counterpart that blocks the entire event loop until the I/O completes. In a server handling 100 concurrent requests, one readFileSync stalls all of them. 

import { readFileSync } from 'node:fs';

// Acceptable: one-off CLI script, no concurrency
const pkg = JSON.parse(readFileSync('package.json', 'utf8'));
console.log(pkg.version);
The rule: If the process serves more than one thing at a time (HTTP server, WebSocket handler, queue consumer), use the async promise API — always. Sync methods are acceptable only in build scripts, CLI tools, and one-shot utilities.

The Decision Flowchart

When you reach for fs, ask yourself:

  1. Is this server code handling concurrent requests? → node:fs/promises, never sync.
  2. Is the file large or of unknown size? → Streams (createReadStream + pipeline). We'll cover this in Lesson 3.
  3. Do I need byte-level positional access? → File handles (open/read/close). Lesson 4.
  4. Is this a CLI/build script that runs and exits? → Sync methods are fine for simplicity.

Try It: Refactor Your app.js

Open app.js and refactor it from the callback API to node:fs/promises. Here's the target: 

import { mkdir } from 'node:fs/promises';

// recursive: true means no error if it already exists (like mkdir -p)
await mkdir('src', { recursive: true });
console.log('src created');

await mkdir('src/components', { recursive: true });
console.log('src/components created');

Run it with:

node app.js

Notice: with { recursive: true }, you don't need to create src before src/components. A single mkdir('src/components', { recursive: true }) creates both levels. This mirrors mkdir -p in the shell.
