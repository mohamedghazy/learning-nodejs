Lesson 03

Reading & Writing Files

Encoding, Buffer vs String, and error handling by error.code — the patterns you'll use daily in full-stack apps.

The Two Return Types of readFile

readFile does exactly one thing differently depending on whether you pass an encoding:

CallReturnsUse For
readFile(path, 'utf8') String — decoded text JSON, config files, HTML, CSS, logs
readFile(path) Buffer — raw bytes Images, PDFs, audio, binaries, encrypted data

This is the single most important decision you make when reading a file: am I dealing with text or binary data? 

import { readFile } from 'node:fs/promises';

// Text: pass encoding → get a string
const json = await readFile('package.json', 'utf8');
console.log(typeof json); // 'string'
console.log(json.length); // character count

// Binary: omit encoding → get a Buffer
const image = await readFile('logo.png');
console.log(Buffer.isBuffer(image)); // true
console.log(image.length);           // byte count

What Is a Buffer? (The Frontend Analogy)

Think of a Buffer as a fixed-length Uint8Array — because that's literally what it is. In the browser, when you use fetch to download an image and call response.arrayBuffer(), you get an ArrayBuffer — raw bytes. Node's Buffer is the same concept, with extra convenience methods for encoding/decoding.

ConceptBrowser EquivalentNode.js
Raw bytes ArrayBuffer / Uint8Array Buffer (subclass of Uint8Array)
Decode bytes → string new TextDecoder().decode(bytes) buffer.toString('utf8')
Encode string → bytes new TextEncoder().encode(str) Buffer.from(str, 'utf8')
Key insight: buffer.length is byte count, not character count. The string "café" is 4 characters but 5 bytes in UTF-8 (the é takes 2 bytes). This distinction matters when you pre-allocate buffers or set Content-Length headers. Use Buffer.byteLength(str, 'utf8') to get the true byte size of a string.

Encodings You'll Actually Use

EncodingWhenExample
'utf8' Text files (JSON, HTML, logs, configs) readFile(path, 'utf8')
'base64' Embedding binary in JSON/URLs buffer.toString('base64')
'hex' Debugging bytes, hash digests hash.digest('hex')
'latin1' Legacy 1-byte-per-char protocols Rarely needed in new code
(none — omit encoding) Binary data: images, audio, PDFs readFile(path) → Buffer

For 99% of full-stack work, you'll use 'utf8' for text and omit encoding for binary. The others show up at specific boundaries (e.g., converting an uploaded image to base64 for a JSON API response).

Writing Files: writeFile, appendFile, truncate

writeFile — Create or Overwrite

import { writeFile } from 'node:fs/promises';

const user = { name: 'Mohamed', role: 'fullstack' };

// Creates the file if missing, completely replaces if it exists
await writeFile(
  'user.json',
  JSON.stringify(user, null, 2),
  'utf8'
);

appendFile — Add Without Replacing

import { appendFile } from 'node:fs/promises';

// Adds to end of file; creates it if missing
const entry = `${new Date().toISOString()} — user created\n`;
await appendFile('events.log', entry, 'utf8');

When to use which:

OperationMethodBehavior
Save config/data writeFile Replace entire contents
Append log entry appendFile Add to end, keep existing
Trim a file truncate(path, n) Keep first n bytes only

Error Handling: Branch on error.code

File operations fail with system errors. Every error carries a .code property — a short string identifying what went wrong at the OS level. Never parse error messages. Always branch on error.code.

The Error Codes You'll Handle Most

CodeMeaningTypical Scenario
ENOENT No such file or directory File doesn't exist, typo in path
EACCES Permission denied Wrong file permissions, running as wrong user
EISDIR Is a directory Called readFile on a directory path
EEXIST Already exists mkdir without recursive: true
ENOSPC No space left on device Disk full during write
EMFILE Too many open files Leaked file handles, no close() in finally
ENOTDIR Not a directory readdir on a file path

The Pattern: try/catch + switch on code

import { readFile } from 'node:fs/promises';

async function loadConfig(path) {
  try {
    const raw = await readFile(path, 'utf8');
    return JSON.parse(raw);
  } catch (error) {
    switch (error.code) {
      case 'ENOENT':
        console.warn(`Config not found at ${path}, using defaults`);
        return { port: 3000, host: 'localhost' };
      case 'EACCES':
        throw new Error(`Permission denied reading ${path}`);
      default:
        // SyntaxError from JSON.parse won't have .code
        // Re-throw unexpected errors — don't swallow them
        throw error;
    }
  }
}
Critical rule: Always re-throw errors you don't specifically handle. Swallowing ENOSPC (disk full) or EMFILE (too many files) silently is a common production failure mode — the real problem stays hidden while your app produces wrong results.

The "Check Then Act" Race Condition

A common mistake: check if a file exists with stat or access, then read it. Between the check and the read, another process can delete the file. This is called a TOCTOU (Time-of-check-to-time-of-use) race condition. 

// ❌ WRONG: race condition
import { stat, readFile } from 'node:fs/promises';

try {
  await stat('config.json'); // file exists now...
} catch {
  // doesn't exist, use defaults
}
// ...but by now it might be deleted!
const data = await readFile('config.json', 'utf8'); // can still fail

// ✅ RIGHT: just read and handle the error
try {
  const data = await readFile('config.json', 'utf8');
  // process data
} catch (err) {
  if (err.code === 'ENOENT') {
    // file doesn't exist — handle it
  } else {
    throw err;
  }
}
The principle: Don't ask "does this file exist?" — just try to use it and handle the failure. This is called EAFP (Easier to Ask Forgiveness than Permission) and it eliminates TOCTOU races.

Real-World Pattern: Safe JSON Config Loader

Here's a production-quality config loader you'd use in a Node.js server or Next.js API route: 

import { readFile, writeFile } from 'node:fs/promises';
import { join } from 'node:path';

const CONFIG_PATH = join(process.cwd(), 'config.json');

const DEFAULTS = {
  port: 3000,
  host: 'localhost',
  logLevel: 'info',
};

export async function loadConfig() {
  try {
    const raw = await readFile(CONFIG_PATH, 'utf8');
    const parsed = JSON.parse(raw);
    // Merge with defaults so missing keys don't break the app
    return { ...DEFAULTS, ...parsed };
  } catch (error) {
    if (error.code === 'ENOENT') {
      // No config file — create one with defaults
      await writeFile(
        CONFIG_PATH,
        JSON.stringify(DEFAULTS, null, 2),
        'utf8'
      );
      return DEFAULTS;
    }
    if (error instanceof SyntaxError) {
      console.error('Config file has invalid JSON:', error.message);
      return DEFAULTS;
    }
    throw error; // EACCES, ENOSPC — let it crash visibly
  }
}

export async function saveConfig(config) {
  await writeFile(
    CONFIG_PATH,
    JSON.stringify(config, null, 2),
    'utf8'
  );
}

Notice the patterns:

  • EAFP: Just read, handle ENOENT if it fails
  • Separate error types: error.code for fs errors, instanceof SyntaxError for JSON parse errors
  • Merge with defaults: Missing config keys don't crash the app
  • Re-throw unknowns: Don't silently swallow permission or disk errors

Writing Binary Data

When handling file uploads or generating binary content (images, PDFs), you work with Buffers: 

import { writeFile, readFile } from 'node:fs/promises';

// Writing binary — pass a Buffer, no encoding needed
const imageBuffer = await getUploadedImage(); // Buffer from request body
await writeFile('uploads/avatar.png', imageBuffer);

// Reading binary → convert to base64 for JSON response
const avatar = await readFile('uploads/avatar.png');
const base64 = avatar.toString('base64');
const dataUrl = `data:image/png;base64,${base64}`;

The Content-Length Gotcha

When setting HTTP headers, you need byte length, not string length: 

const body = JSON.stringify({ message: 'مرحبا' }); // Arabic text

// ❌ WRONG: string.length counts characters (some are multi-byte)
res.setHeader('Content-Length', body.length); // WRONG!

// ✅ RIGHT: Buffer.byteLength counts actual bytes
res.setHeader('Content-Length', Buffer.byteLength(body, 'utf8'));
This matters with non-ASCII text. Arabic, Chinese, Japanese, emoji — all take multiple bytes per character in UTF-8. If your Content-Length is wrong, the client truncates or hangs waiting for more bytes.
