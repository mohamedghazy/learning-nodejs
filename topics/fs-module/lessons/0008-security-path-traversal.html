Lesson 08

Security — Path Traversal Prevention

The #1 file system vulnerability in web apps, and the 5-line pattern that stops it.

The Attack

Your API serves files from an uploads/ directory. A user requests their avatar: 

GET /files/avatar.png
→ server reads: uploads/avatar.png ✓

An attacker requests:

GET /files/../../../etc/passwd
→ server reads: uploads/../../../etc/passwd
→ resolves to: /etc/passwd ✗ (entire system exposed!)

This is a path traversal (or directory traversal) attack. By injecting ../ sequences, an attacker escapes your intended directory and reads (or writes) arbitrary files on your server. It's in the OWASP Top 10 and remains one of the most common server vulnerabilities.

This affects you directly. If you build any API that serves user-uploaded files, generates download links from user input, or reads files based on URL parameters — you need this protection. Express static file serving, Next.js API routes handling uploads, any file-serving endpoint.

Why Naive Approaches Fail

❌ Checking for .. in the input string

// ❌ WRONG: easily bypassed
if (userInput.includes('..')) {
  throw new Error('Invalid path');
}

// Bypasses:
// - URL encoding: %2e%2e%2f  (../ encoded)
// - Double encoding: %252e%252e%252f
// - Null bytes: ../\0.png (older systems)
// - Windows: ..\\ instead of ../

❌ Using path.join alone

// ❌ WRONG: join normalizes but doesn't constrain
const filePath = join('uploads', userInput);
// join('uploads', '../../etc/passwd')
// → '../etc/passwd' — still escapes!

❌ Using path.normalize alone

// ❌ WRONG: normalize resolves .., but doesn't check boundaries
const normalized = path.normalize(userInput);
// Still allows escape if you don't verify the result

The Correct Pattern (5 Lines)

The fix: resolve the full path, then verify it starts with your base directory. 

import { resolve, sep } from 'node:path';
import { readFile } from 'node:fs/promises';

const BASE_DIR = resolve('uploads'); // absolute path to allowed directory

function safePath(userInput) {
  const target = resolve(BASE_DIR, userInput);

  // Verify the resolved path is still inside BASE_DIR
  if (!target.startsWith(BASE_DIR + sep) && target !== BASE_DIR) {
    throw new Error('Path traversal blocked');
  }

  return target;
}

// Safe usage:
const file = await readFile(safePath('avatar.png'));        // ✓ OK
const attack = await readFile(safePath('../../etc/passwd')); // ✗ Throws!

Why this works:

  1. resolve(BASE_DIR, userInput) normalizes all .., ., double separators, etc. into a clean absolute path
  2. target.startsWith(BASE_DIR + sep) verifies the result is still inside the allowed directory
  3. The + sep prevents a sibling directory bypass — without it, /app/uploads-private/secret.key would pass a bare startsWith('/app/uploads') check

The + sep Guard Explained

This is the subtle detail most developers miss: 

const BASE_DIR = '/app/uploads';

// Without + sep:
'/app/uploads-private/secret.key'.startsWith('/app/uploads')
// → true! Sibling directory passes the check!

// With + sep:
'/app/uploads-private/secret.key'.startsWith('/app/uploads/')
// → false ✓ Correctly blocked

'/app/uploads/avatar.png'.startsWith('/app/uploads/')
// → true ✓ Correctly allowed

Real-World Pattern: Express File Download Endpoint

import express from 'express';
import { resolve, sep } from 'node:path';
import { createReadStream } from 'node:fs';
import { stat } from 'node:fs/promises';
import { pipeline } from 'node:stream/promises';

const app = express();
const UPLOADS_DIR = resolve('uploads');

app.get('/files/:filename', async (req, res) => {
  try {
    // 1. Validate path
    const target = resolve(UPLOADS_DIR, req.params.filename);
    if (!target.startsWith(UPLOADS_DIR + sep)) {
      return res.status(400).json({ error: 'Invalid filename' });
    }

    // 2. Check file exists and is a file (not a directory)
    const info = await stat(target);
    if (!info.isFile()) {
      return res.status(400).json({ error: 'Not a file' });
    }

    // 3. Stream the file (constant memory)
    res.setHeader('Content-Length', info.size);
    await pipeline(createReadStream(target), res);
  } catch (err) {
    if (err.code === 'ENOENT') {
      res.status(404).json({ error: 'File not found' });
    } else {
      res.status(500).json({ error: 'Internal error' });
    }
  }
});

Notice how this combines multiple lessons:

  • Lesson 4 — streams + pipeline for serving the file efficiently
  • Lesson 3 — error handling by error.code
  • Lesson 6 — resolve for building absolute paths
  • This lesson — startsWith(BASE + sep) for security

Real-World Pattern: Next.js API Route With Upload Validation

// app/api/files/[name]/route.ts
import { resolve, sep } from 'node:path';
import { readFile } from 'node:fs/promises';
import { NextResponse } from 'next/server';

const UPLOADS_DIR = resolve(process.cwd(), 'uploads');

export async function GET(
  request: Request,
  { params }: { params: { name: string } }
) {
  const target = resolve(UPLOADS_DIR, params.name);

  // Path traversal check
  if (!target.startsWith(UPLOADS_DIR + sep)) {
    return NextResponse.json(
      { error: 'Invalid filename' },
      { status: 400 }
    );
  }

  try {
    const file = await readFile(target);
    return new NextResponse(file, {
      headers: { 'Content-Type': 'application/octet-stream' },
    });
  } catch (err) {
    if (err.code === 'ENOENT') {
      return NextResponse.json({ error: 'Not found' }, { status: 404 });
    }
    throw err;
  }
}

Additional Security Checks

Path traversal prevention is the baseline. In production, also consider:

CheckWhyHow
Filename sanitization Remove special chars that could cause issues Strip anything except [a-zA-Z0-9._-]
File type validation Don't serve executable code as downloads Magic number check (Lesson 5)
Size limits Prevent resource exhaustion Check stat.size before reading/streaming
Symlink resolution Symlinks can escape the directory Use realpath() then re-check boundary
Rate limiting Prevent directory enumeration attacks Limit requests per IP to file endpoints

Symlink-Aware Validation

import { realpath } from 'node:fs/promises';
import { resolve, sep } from 'node:path';

async function safePathStrict(baseDir, userInput) {
  const target = resolve(baseDir, userInput);

  // First check: resolved path is within base
  if (!target.startsWith(baseDir + sep) && target !== baseDir) {
    throw new Error('Path traversal blocked');
  }

  // Second check: follow symlinks and verify real path is still within base
  const real = await realpath(target);
  if (!real.startsWith(baseDir + sep) && real !== baseDir) {
    throw new Error('Symlink escape blocked');
  }

  return real;
}

Filename Sanitization Helper

function sanitizeFilename(input) {
  // Remove path separators and special characters
  // Allow only alphanumeric, dots, hyphens, underscores
  return input
    .replace(/[^a-zA-Z0-9._-]/g, '')
    .replace(/^\.+/, ''); // Remove leading dots (hidden files)
}

// Usage with path traversal check (defense in depth)
const filename = sanitizeFilename(req.params.name);
const target = safePath(filename); // still validate!
Defense in depth: Sanitize the input AND validate the resolved path. Either check alone might have edge cases. Together, they're robust.

Why do we check startsWith(BASE_DIR + sep) instead of just startsWith(BASE_DIR)?

Security Checklist for File-Serving Endpoints

  1. ☐ Path traversal check: resolve() + startsWith(base + sep)
  2. ☐ Filename sanitization: strip special characters
  3. ☐ Symlink check: realpath() + re-verify boundary
  4. ☐ File type validation: magic numbers for uploads
  5. ☐ Size limits: reject oversized requests
  6. ☐ Error responses: don't leak internal paths in error messages
  7. ☐ Rate limiting: prevent directory enumeration
