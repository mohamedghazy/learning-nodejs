Lesson 05

File Handles & Byte-Level Access

When you need to read 16 bytes at offset 1024, or write to a specific position without touching the rest — file handles are the tool.

When File Handles Make Sense

You now know two approaches: readFile (whole file into memory) and streams (chunk-by-chunk). File handles are the third option — the lowest-level one. They give you a direct connection to the file where you control exactly what bytes to read, from where, and how many.

ApproachControl LevelBest For
readFile / writeFile Whole file at once Small files, JSON configs
Streams + pipeline Sequential chunks Large files, transforms, HTTP responses
File handles Specific bytes at specific positions Binary headers, random access, partial updates

Real-world scenarios for file handles:

  • Reading file headers — detect file type by reading first 4–16 bytes (magic numbers)
  • Binary file formats — WAV audio, PNG images, database files with fixed-size records
  • Partial updates — modify a specific record in a file without rewriting everything
  • Resume downloads — write data at a specific byte offset for interrupted transfers
  • Memory-mapped-like patterns — read only what you need from a massive file
For most full-stack work, you won't need file handles. readFile and streams cover 95% of cases. But understanding handles makes you confident with binary formats, file validation, and those rare cases where only positional access works.

The Lifecycle: Open → Read/Write → Close

A file handle is a low-level resource. You must manage its lifecycle:

  1. Open — await open(path, flags) → returns a FileHandle
  2. Use — handle.read(), handle.write(), handle.stat()
  3. Close — await handle.close() in a finally block
import { open } from 'node:fs/promises';

let handle;
try {
  handle = await open('data.bin', 'r');
  // ... use handle ...
} finally {
  await handle?.close();
}
Always close in finally. Forgetting to close leaks file descriptors. Enough leaks → EMFILE (too many open files) → your server can't open any new files, connections, or sockets. This is a common production incident.

File Open Flags

The second argument to open() specifies what you plan to do with the file:

FlagMeaningCreates if missing?Truncates?
'r'Read onlyNo (throws ENOENT)No
'r+'Read + writeNo (throws ENOENT)No
'w'Write onlyYesYes (empties file)
'w+'Read + writeYesYes (empties file)
'a'Append onlyYesNo
'a+'Read + appendYesNo

The most common for reading binary headers: 'r'. For writing at specific positions: 'r+' (reads and writes without truncating the existing content).

Reading at a Specific Position

handle.read(buffer, offset, length, position) fills your pre-allocated buffer:

  • buffer — a Buffer to fill with data
  • offset — where in the buffer to start writing (usually 0)
  • length — how many bytes to read
  • position — where in the file to start reading (byte offset)
import { open } from 'node:fs/promises';

async function readFileHeader(path) {
  let handle;
  try {
    handle = await open(path, 'r');

    // Read the first 16 bytes (common header size)
    const header = Buffer.alloc(16);
    const { bytesRead } = await handle.read(header, 0, 16, 0);

    console.log(`Read ${bytesRead} bytes`);
    console.log('Header (hex):', header.toString('hex'));
    console.log('Header (ascii):', header.toString('ascii'));

    return header;
  } finally {
    await handle?.close();
  }
}

await readFileHeader('photo.png');
// Header (hex): 89504e470d0a1a0a0000000d49484452
// The first 8 bytes of any PNG: 89 50 4e 47 0d 0a 1a 0a

Real-World Pattern: File Type Detection (Magic Numbers)

Every binary file format starts with a "magic number" — specific bytes that identify the format. This is how servers validate uploaded files (you can't trust the filename extension): 

import { open } from 'node:fs/promises';

const MAGIC_NUMBERS = {
  png:  Buffer.from([0x89, 0x50, 0x4E, 0x47]),
  jpg:  Buffer.from([0xFF, 0xD8, 0xFF]),
  gif:  Buffer.from([0x47, 0x49, 0x46]),
  pdf:  Buffer.from([0x25, 0x50, 0x44, 0x46]),
  zip:  Buffer.from([0x50, 0x4B, 0x03, 0x04]),
};

async function detectFileType(path) {
  let handle;
  try {
    handle = await open(path, 'r');
    const header = Buffer.alloc(4);
    await handle.read(header, 0, 4, 0);

    for (const [type, magic] of Object.entries(MAGIC_NUMBERS)) {
      if (header.subarray(0, magic.length).equals(magic)) {
        return type;
      }
    }
    return 'unknown';
  } finally {
    await handle?.close();
  }
}

console.log(await detectFileType('upload.png')); // 'png'
console.log(await detectFileType('resume.pdf')); // 'pdf'
Why this matters for full-stack: When handling file uploads, never trust the Content-Type header or file extension — both can be faked. Read the first few bytes to verify the actual file format. This is a security pattern you'll use in Express/Fastify upload handlers.

Reading in Chunks (Manual Streaming)

You can build your own chunked reader with file handles. This is useful when you need something streams don't offer — like reading chunks from specific offsets: 

import { open } from 'node:fs/promises';

async function readInChunks(path) {
  let handle;
  try {
    handle = await open(path, 'r');
    const chunkSize = 64 * 1024; // 64KB per read
    const buffer = Buffer.alloc(chunkSize);
    let position = 0;
    let totalBytes = 0;

    while (true) {
      const { bytesRead } = await handle.read(
        buffer, 0, chunkSize, position
      );
      if (bytesRead === 0) break; // EOF

      // Process only the bytes actually read
      const chunk = buffer.subarray(0, bytesRead);
      // ... do something with chunk ...

      position += bytesRead;
      totalBytes += bytesRead;
    }

    console.log(`Read ${totalBytes} bytes total`);
  } finally {
    await handle?.close();
  }
}

Note: for most sequential reading, streams are simpler. Use file handles for chunked reading only when you need positional control (jumping to specific offsets).

Writing at a Specific Position

handle.write(buffer, offset, length, position) writes bytes at an exact file offset: 

import { open } from 'node:fs/promises';

async function patchFile(path, data, fileOffset) {
  let handle;
  try {
    // 'r+' = read/write without truncating
    handle = await open(path, 'r+');
    const buffer = Buffer.from(data, 'utf8');
    const { bytesWritten } = await handle.write(
      buffer, 0, buffer.length, fileOffset
    );
    console.log(`Wrote ${bytesWritten} bytes at offset ${fileOffset}`);
  } finally {
    await handle?.close();
  }
}

// Overwrite bytes 10-14 with "HELLO"
await patchFile('data.bin', 'HELLO', 10);
Important: Writing at a position overwrites the bytes there — it doesn't insert. If your file has AABBCCDD and you write XX at position 2, you get AAXXCCDD, not AAXXBBCCDD. Files don't support insert-in-the-middle — you'd have to rewrite from that point forward.

Real-World Pattern: Reading WAV Audio Duration

WAV files have a fixed header structure. You can read the duration without loading the audio data — just read the header bytes: 

import { open } from 'node:fs/promises';

async function getWavDuration(path) {
  let handle;
  try {
    handle = await open(path, 'r');
    const header = Buffer.alloc(44); // WAV header is 44 bytes
    await handle.read(header, 0, 44, 0);

    // Verify it's a WAV file
    const riff = header.toString('ascii', 0, 4);
    const wave = header.toString('ascii', 8, 12);
    if (riff !== 'RIFF' || wave !== 'WAVE') {
      throw new Error('Not a WAV file');
    }

    // Extract fields using little-endian reads
    const byteRate = header.readUInt32LE(28);  // bytes per second
    const dataSize = header.readUInt32LE(40); // size of audio data

    const durationSec = dataSize / byteRate;
    console.log(`Duration: ${durationSec.toFixed(2)}s`);
    return durationSec;
  } finally {
    await handle?.close();
  }
}

This reads only 44 bytes regardless of whether the audio file is 1MB or 1GB. That's the power of positional access — you skip the data you don't need.

The FileHandle API Surface

A FileHandle object gives you these methods:

MethodWhat It Does
handle.read(buffer, offset, length, position)Read bytes into buffer from file position
handle.write(buffer, offset, length, position)Write bytes from buffer to file position
handle.stat()Get file metadata (size, dates, etc.)
handle.truncate(len)Shrink/grow file to exact size
handle.close()Release the file descriptor
handle.readFile(options)Convenience: read entire file (like fs.readFile)
handle.writeFile(data)Convenience: write entire file (like fs.writeFile)
handle.createReadStream()Create a readable stream from this handle
handle.createWriteStream()Create a writable stream from this handle

Notice: FileHandle can also create streams and do whole-file operations. This lets you mix approaches — open a handle, read the header manually, then create a stream for the rest.

The AbortController Pattern

For long reads that might need cancellation (e.g., user navigates away, request times out): 

import { readFile } from 'node:fs/promises';

const controller = new AbortController();

// Cancel after 5 seconds
setTimeout(() => controller.abort(), 5000);

try {
  const data = await readFile('slow-network-file.bin', {
    signal: controller.signal,
  });
} catch (err) {
  if (err.name === 'AbortError') {
    console.log('Read cancelled — took too long');
  } else {
    throw err;
  }
}

AbortSignal is supported by readFile and fs.watch. You already know AbortController from the Fetch API in the browser — same pattern here.

Hands-On Exercise

Create a script called detect-type.js that reads the first 4 bytes of a file and detects whether it's a PNG, JPEG, PDF, or ZIP: 

// Usage: node detect-type.js myfile.png
// Output: "myfile.png is a PNG file"

Steps:

  1. Get the filename from process.argv[2]
  2. Open the file with open(path, 'r')
  3. Allocate a 4-byte Buffer
  4. Read 4 bytes at position 0
  5. Compare against known magic numbers
  6. Close the handle in finally

Test with files you already have, or create test files:

# Download a test PNG
curl -o test.png https://www.google.com/favicon.ico
node detect-type.js test.png
