Lesson 07

Concurrency Patterns

Promise.all vs Promise.allSettled — when one failure should abort everything, and when it shouldn't.

The Problem: Sequential Is Slow

You need to read 10 config files. Each read takes ~10ms (disk I/O). If you await them one by one, that's 100ms minimum. But disk I/O is non-blocking — Node can start all 10 reads simultaneously and wait for them all to finish: 

import { readFile } from 'node:fs/promises';

// ❌ Sequential: ~100ms (waits for each before starting next)
const a = await readFile('a.json', 'utf8');
const b = await readFile('b.json', 'utf8');
const c = await readFile('c.json', 'utf8');

// ✅ Concurrent: ~10ms (all reads happen simultaneously)
const [a, b, c] = await Promise.all([
  readFile('a.json', 'utf8'),
  readFile('b.json', 'utf8'),
  readFile('c.json', 'utf8'),
]);

This isn't parallelism (Node is single-threaded) — it's concurrency. The I/O operations happen at the OS level simultaneously while your JavaScript waits. You already do this with fetch in React — same concept, different I/O.

The Decision: All-or-Nothing vs Partial Success

MethodBehavior on failureUse When
Promise.all() Rejects immediately on first failure All operations must succeed — one failure means abort
Promise.allSettled() Waits for all to complete, reports each result Partial failure is acceptable — process what succeeded

Promise.all — All Must Succeed

Use when every operation is required. If any single promise rejects, Promise.all rejects immediately with that error — the other operations still complete in the background but their results are discarded.

Example: Load required configuration files

import { readFile } from 'node:fs/promises';

const configFiles = [
  'config/database.json',
  'config/auth.json',
  'config/features.json',
];

try {
  // Start all reads simultaneously, parse each as JSON
  const [dbConfig, authConfig, featuresConfig] = await Promise.all(
    configFiles.map(file =>
      readFile(file, 'utf8').then(JSON.parse)
    )
  );

  console.log('All configs loaded:', { dbConfig, authConfig, featuresConfig });
} catch (error) {
  // If ANY file fails, we land here
  console.error(`Cannot start: ${error.message}`);
  process.exit(1);
}

Why all-or-nothing here? If the database config is missing, the server can't start. There's no point loading the other configs — we need all of them.

Example: Write multiple related files

import { writeFile, mkdir } from 'node:fs/promises';

async function generateReports(data) {
  await mkdir('reports', { recursive: true });

  const reports = [
    { path: 'reports/daily.json', content: data.daily },
    { path: 'reports/weekly.json', content: data.weekly },
    { path: 'reports/monthly.json', content: data.monthly },
  ];

  // Write all concurrently — if disk is full, all fail together
  await Promise.all(
    reports.map(r =>
      writeFile(r.path, JSON.stringify(r.content, null, 2), 'utf8')
    )
  );
}

Promise.allSettled — Tolerate Partial Failure

Use when some operations can fail without invalidating the whole batch. Promise.allSettled always resolves (never rejects). It returns an array where each entry is either:

  • { status: 'fulfilled', value: ... } — success
  • { status: 'rejected', reason: ... } — failure

Example: Load optional plugin configs

import { readFile } from 'node:fs/promises';

const pluginConfigs = [
  'plugins/analytics.json',
  'plugins/logging.json',
  'plugins/caching.json',    // might not exist
  'plugins/monitoring.json',  // might not exist
];

const results = await Promise.allSettled(
  pluginConfigs.map(file =>
    readFile(file, 'utf8').then(JSON.parse)
  )
);

// Separate successes from failures
const loaded = results
  .filter(r => r.status === 'fulfilled')
  .map(r => r.value);

const failed = results
  .filter(r => r.status === 'rejected')
  .map((r, i) => ({ file: pluginConfigs[i], code: r.reason.code }));

console.log(`Loaded ${loaded.length} plugins, ${failed.length} unavailable`);
// → "Loaded 2 plugins, 2 unavailable"

if (failed.length > 0) {
  console.warn('Missing plugins:', failed);
}

Why tolerate failure here? Plugins are optional. The server should start even if some plugin configs are missing — just without those plugins.

Example: Process user uploads (best effort)

import { readFile, writeFile } from 'node:fs/promises';
import { join } from 'node:path';

async function processUploads(filePaths) {
  const results = await Promise.allSettled(
    filePaths.map(async (path) => {
      const data = await readFile(path);
      const processed = await processImage(data);
      const outPath = join('processed', basename(path));
      await writeFile(outPath, processed);
      return outPath;
    })
  );

  const succeeded = results.filter(r => r.status === 'fulfilled');
  const failed = results.filter(r => r.status === 'rejected');

  return {
    processed: succeeded.map(r => r.value),
    errors: failed.map((r, i) => ({
      file: filePaths[i],
      error: r.reason.message,
    })),
  };
}

//
//   YES → Promise.allSettled()
//         Examples: optional plugins, batch processing, user uploads
//
//   NO  → Promise.all()
//         Examples: required configs, database + cache init, related writes

Advanced: Controlled Concurrency

Promise.all with 10,000 file reads will try to open 10,000 file descriptors simultaneously — hitting EMFILE (too many open files). You need a concurrency limit: 

async function mapWithConcurrency(items, fn, limit = 10) {
  const results = [];
  const executing = new Set();

  for (const [index, item] of items.entries()) {
    const promise = fn(item, index).then(result => {
      executing.delete(promise);
      return result;
    });
    executing.add(promise);
    results.push(promise);

    if (executing.size >= limit) {
      await Promise.race(executing);
    }
  }

  return Promise.all(results);
}

// Process 1000 files, max 20 at a time
const files = await readdir('uploads');
const processed = await mapWithConcurrency(
  files,
  async (file) => {
    const data = await readFile(join('uploads', file));
    return processFile(data);
  },
  20 // max 20 concurrent reads
);
Production tip: For simple cases, the npm package p-limit does this in a few lines. But the pattern above shows what's happening under the hood — it uses Promise.race to wait for any one slot to free up before starting the next operation.

Pattern: Parallel Reads, Sequential Processing

Sometimes you want to read files concurrently (fast I/O) but process results in order: 

import { readFile } from 'node:fs/promises';

const files = ['chapter1.md', 'chapter2.md', 'chapter3.md'];

// Start all reads concurrently
const contents = await Promise.all(
  files.map(f => readFile(f, 'utf8'))
);

// Process in order (contents[0] is always chapter1, etc.)
const book = contents.join('\n\n---\n\n');
console.log(`Combined ${contents.length} chapters`);

Promise.all preserves the order of the input array — contents[0] always corresponds to files[0], regardless of which file finished reading first.

Anti-Pattern: await in a Loop

This is the most common performance mistake in Node.js code: 

// ❌ Sequential: each iteration waits for the previous to finish
for (const file of files) {
  const data = await readFile(file, 'utf8'); // blocks the loop!
  results.push(JSON.parse(data));
}

// ✅ Concurrent: all reads start immediately
const results = await Promise.all(
  files.map(file => readFile(file, 'utf8').then(JSON.parse))
);
Exception: await in a loop is correct when operations have dependencies — e.g., each iteration needs the result of the previous one, or you're deliberately limiting concurrency. But for independent I/O operations (reading unrelated files), it's always a mistake.
