Events — Lesson 02

The Event Loop — 6 Phases

How Node.js handles thousands of concurrent operations on a single thread. The loop that makes non-blocking I/O work.

The Big Picture

Node.js runs your JavaScript on a single thread. But it handles thousands of concurrent connections. How? The event loop — a continuous cycle that checks for completed I/O and runs the appropriate callbacks.

The model:

  1. Your code starts an async operation (read file, network request, timer)
  2. Node hands it to libuv (C library that manages a thread pool + OS kernel calls)
  3. Your code continues executing (non-blocking)
  4. When the operation completes, libuv queues the callback
  5. The event loop picks up the callback and runs it on the main thread
Frontend analogy: This is exactly how the browser event loop works with setTimeout, fetch, and DOM events. You already understand this model — Node.js just adds more I/O types (files, network, child processes).

The 6 Phases

The event loop runs in a fixed order, cycling through these phases:

   ┌───────────────────────────┐
┌─>│         timers             │ ← setTimeout, setInterval
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks      │ ← I/O errors, OS callbacks
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare        │ ← internal use only
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           poll             │ ← I/O callbacks (fs, network)
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │           check            │ ← setImmediate
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │      close callbacks       │ ← socket.on('close')
│  └─────────────┬─────────────┘
│                 │
└─────────────────┘ (loop back to timers)

Microtasks: Between Every Phase

Between every phase (and after every callback), Node.js drains two microtask queues:

  1. process.nextTick() callbacks — highest priority, runs first
  2. Promise .then()/.catch()/.finally() callbacks — runs second
setTimeout(() => console.log('1: timer'), 0);
setImmediate(() => console.log('2: immediate'));
Promise.resolve().then(() => console.log('3: promise'));
process.nextTick(() => console.log('4: nextTick'));
console.log('5: sync');

// Output:
// 5: sync       (synchronous — runs first)
// 4: nextTick   (microtask — highest priority)
// 3: promise    (microtask — after nextTick)
// 1: timer      (timer phase)
// 2: immediate  (check phase)
Execution order: Sync code → nextTick → Promises → Event loop phases. This is the same as the browser: sync → microtasks → macrotasks. process.nextTick is Node-specific and has no browser equivalent.

The Poll Phase — Where Your Server Lives

The poll phase is where Node spends most of its time. It:

  • Calculates how long to wait for new I/O
  • Runs callbacks for completed I/O (file reads, network data, DNS lookups)
  • If no I/O is pending and no timers are due, it blocks here waiting

This is why an idle HTTP server uses almost zero CPU — it's blocking in the poll phase waiting for incoming connections.

setTimeout vs setImmediate vs process.nextTick

APIWhen it runsUse for
setTimeout(fn, 0) Next timers phase (minimum ~1ms delay) Delay execution to next loop iteration
setImmediate(fn) Check phase of current/next iteration Run after I/O events are processed
process.nextTick(fn) Before the event loop continues (microtask) Run ASAP, before any I/O or timers
⚠️ process.nextTick can starve I/O. If you recursively call process.nextTick, the event loop never advances to the poll phase — I/O callbacks never fire. Prefer setImmediate for deferring work that shouldn't block I/O.

Why This Matters for Full-Stack

  • Blocking the event loop = freezing your entire server. One readFileSync in a request handler blocks ALL other requests.
  • CPU-intensive work (image processing, crypto, parsing large JSON) blocks the loop. Solution: Worker Threads or offload to a queue.
  • Promise.resolve().then() runs before timers — this explains why await can sometimes execute before setTimeout(..., 0).
