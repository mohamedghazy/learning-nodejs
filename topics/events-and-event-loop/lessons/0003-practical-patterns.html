Events — Lesson 03

Practical Event Patterns

Memory leaks, typed events, async listeners, and the patterns you'll use in production services.

Memory Leak: The MaxListeners Warning

By default, Node warns if you register more than 10 listeners for a single event. This usually means you're adding listeners in a loop without cleaning up: 

// ❌ Memory leak: adds a new listener on every request
server.on('request', (req, res) => {
  db.on('ready', () => { /* ... */ }); // NEW listener every request!
});

// ✅ Fix: use once() or register outside the loop
server.on('request', (req, res) => {
  db.once('ready', () => { /* auto-removed */ });
});

If you genuinely need more than 10 listeners, increase the limit explicitly: 

emitter.setMaxListeners(50); // I know what I'm doing
// or 0 for unlimited (be careful)

Pattern: Event-Driven Decoupling

Instead of one module directly calling another, emit an event. Other modules subscribe: 

// app-events.js — shared event bus
import { EventEmitter } from 'node:events';
export const appEvents = new EventEmitter();

// auth-service.js
import { appEvents } from './app-events.js';
export function login(user) {
  // ... authenticate ...
  appEvents.emit('user:login', { userId: user.id, time: Date.now() });
}

// analytics.js — listens, doesn't know about auth
import { appEvents } from './app-events.js';
appEvents.on('user:login', ({ userId, time }) => {
  console.log(`Analytics: user ${userId} logged in at ${time}`);
});

// audit-log.js — also listens, also doesn't know about auth
import { appEvents } from './app-events.js';
appEvents.on('user:login', ({ userId }) => {
  console.log(`Audit: login event for ${userId}`);
});

Pattern: Async Event Listeners

emit() doesn't await async listeners. If your listener is async, errors won't propagate unless you handle them: 

// ❌ Error disappears (unhandled rejection)
emitter.on('order:created', async (order) => {
  await sendEmail(order); // if this throws, nobody catches it
});

// ✅ Wrap async listeners with error handling
emitter.on('order:created', async (order) => {
  try {
    await sendEmail(order);
  } catch (err) {
    console.error('Email failed:', err.message);
    // Optionally: emitter.emit('error', err)
  }
});

Pattern: AbortSignal + EventEmitter

Cancel listeners when an operation is aborted:

import { once } from 'node:events';

const controller = new AbortController();

try {
  const [data] = await once(emitter, 'data', { signal: controller.signal });
  console.log('Got data:', data);
} catch (err) {
  if (err.code === 'ABORT_ERR') {
    console.log('Cancelled waiting for event');
  }
}

// Later: controller.abort() to stop waiting

Pattern: Typed Events (TypeScript)

In TypeScript projects, type your events for safety:

import { EventEmitter } from 'node:events';

interface OrderEvents {
  'order:created': [order: { id: string; total: number }];
  'order:cancelled': [orderId: string];
  'error': [error: Error];
}

class OrderService extends (EventEmitter as {
  new(): TypedEmitter<OrderEvents>
}) {
  // Now emit() and on() are type-safe
}

When NOT to Use EventEmitter

  • Simple function calls — if there's only one subscriber, just call the function directly
  • Request/response patterns — events are fire-and-forget; if you need a return value, use async functions
  • Cross-service communication — use a message queue (Redis, RabbitMQ) instead of in-process events
