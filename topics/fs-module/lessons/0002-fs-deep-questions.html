Lesson 02

Three Deep Questions

Why the callback API still exists, how fs relates to Next.js, and the real edge cases of top-level await.

1. Why Does the Callback API Still Exist?

If node:fs/promises is the modern default, why hasn't Node.js deprecated the callback API? Three reasons — historical, practical, and political.

Historical: The Ryan Dahl Decision (2010)

In February 2010, Ryan Dahl (Node's creator) made a deliberate choice to use the callback(err, result) pattern instead of Promises for all core APIs. His reasoning: Promises are a "higher-level construct" that belongs in userland, not in the core runtime. This was before ES2015 standardized Promises, so there was no agreed-upon Promise implementation anyway. Every core module — fs, http, net, crypto — was built around callbacks. (source)

Practical: Backward Compatibility

Millions of packages on npm depend on the callback API. Deprecating it would break the entire ecosystem. Node.js follows semver — removing a stable API requires a major version bump and years of deprecation warnings. The callback API is stable (not deprecated), so it won't disappear.

Performance: A Marginal (But Real) Difference

The promise API wraps callbacks internally — every fs/promises call creates a Promise object, which has a small allocation cost. For the vast majority of use cases, this is negligible. But in extreme hot paths (think: a file-processing pipeline doing millions of tiny reads per second), the callback API avoids that per-call object allocation. This is why the Node.js docs once noted that "the callback-based versions are preferable for maximum performance." In practice, you'll almost never hit this threshold.

Your takeaway: Use node:fs/promises for all new code. The callback API exists for backward compatibility and niche performance scenarios — not because it's better. If you encounter callback-style fs code in a codebase, you'll understand it, but don't write new code that way.

When You Might Still See Callbacks

  • Legacy codebases — anything written before Node 14 probably uses callbacks
  • Streams — createReadStream and createWriteStream live on the callback-based node:fs import (not node:fs/promises), though you combine them with pipeline from node:stream/promises
  • fs.watch — the file watcher uses an event emitter pattern (not promises), because it's a long-lived listener, not a one-shot operation

2. How This Relates to Next.js API Routes

As a Next.js developer, you already know the client/server boundary intimately. The fs module maps onto this boundary perfectly:

The Rule: fs Is Server-Only

node:fs is a Node.js built-in that accesses the operating system's file system. Browsers don't have file system access (by design — security). So:

ContextCan use fs?Why
API Routes (app/api/*/route.ts) ✅ Yes Runs on the server (Node.js runtime)
Server Components (app/page.tsx without 'use client') ✅ Yes Rendered on the server
getServerSideProps / getStaticProps (Pages Router) ✅ Yes Server-side data fetching
Client Components ('use client') ❌ No Bundled for the browser — fs doesn't exist there
Middleware (middleware.ts) ❌ No Runs on Edge Runtime (V8 isolate, no Node.js APIs)

The Classic "Module not found: Can't resolve 'fs'" Error

You'll get this error when you accidentally import fs in code that gets bundled for the client. Next.js tree-shakes server-only code, but only when it can prove the import is unreachable from the client bundle. If you import fs at the top of a file that's used by both server and client code, the bundler includes it in the client bundle and fails. 

// ❌ This will break if the file is ever imported client-side
import { readFile } from 'node:fs/promises';
import { SomeComponent } from './SomeComponent';

// ✅ Keep fs usage in dedicated server-only files
// lib/data.server.ts  (the .server suffix is a convention, not required)
import { readFile } from 'node:fs/promises';
import { join } from 'node:path';

export async function getConfig() {
  const raw = await readFile(
    join(process.cwd(), 'config.json'),
    'utf8'
  );
  return JSON.parse(raw);
}

Practical Example: Reading a File in an API Route

// app/api/config/route.ts
import { readFile } from 'node:fs/promises';
import { join } from 'node:path';
import { NextResponse } from 'next/server';

export async function GET() {
  try {
    const filePath = join(process.cwd(), 'data', 'settings.json');
    const data = JSON.parse(
      await readFile(filePath, 'utf8')
    );
    return NextResponse.json(data);
  } catch (error) {
    if (error.code === 'ENOENT') {
      return NextResponse.json(
        { error: 'Settings file not found' },
        { status: 404 }
      );
    }
    throw error;
  }
}
Vercel/Serverless caveat: On serverless platforms, the file system is read-only and ephemeral. You can read files that are part of your deployment bundle (like JSON configs), but you cannot write files that persist between requests. For file uploads or persistent storage, use object storage (S3, R2) or a database. fs.writeFile will "work" but the file vanishes when the function instance is recycled.

Use process.cwd() vs import.meta.dirname

In Next.js, process.cwd() gives you the project root (where package.json lives). Use it for accessing project files. import.meta.dirname (Node 20.11+) gives you the directory of the current file — useful when you need paths relative to a specific module, not the project root.

3. Top-Level Await: The Edge Cases That Bite

Top-level await is available in ES modules (your project already has "type": "module"). It's elegant and powerful, but it has real edges you need to know.

Edge Case 1: It Only Works in ESM

Top-level await is an ES module feature. It does not work in CommonJS files. If your package.json doesn't have "type": "module", or you're in a .cjs file, you'll get a syntax error. 

// package.json must have "type": "module"
// OR the file must be named .mjs
// OTHERWISE: SyntaxError: await is only valid in async functions

const data = await readFile('config.json', 'utf8'); // ✓ in ESM

Edge Case 2: It Blocks Importers (The "Waterfall" Problem)

This is the big one. When a module uses top-level await, any module that imports it must wait until the await resolves. If multiple modules use top-level await, they execute in import order, not in parallel. 

// slow-config.js — takes 2 seconds
export const config = await fetchRemoteConfig(); // 2s

// slow-db.js — takes 1 second
export const db = await connectToDatabase(); // 1s

// main.js — imports both
import { config } from './slow-config.js'; // waits 2s
import { db } from './slow-db.js';         // THEN waits 1s
// Total: 3 seconds — they ran in series, not parallel!

This is the concern Rich Harris flagged in his famous "Top-level await is a footgun" gist (2017). The final TC39 spec (variant B) improved this — sibling imports that don't depend on each other can now execute their top-level awaits concurrently. But sequential imports still serialize.

The fix: Don't put slow async operations at the top level of modules that get imported widely. Instead, export an async initialization function: 
// ✅ Better: export a factory, don't run at import time
import { readFile } from 'node:fs/promises';

let _config;

export async function getConfig() {
  if (!_config) {
    _config = JSON.parse(
      await readFile('config.json', 'utf8')
    );
  }
  return _config;
}

// ✅ When TLA is appropriate: the entry-point script
// server.js (this is the root — nothing imports it)
const config = await getConfig();
const db = await connectToDatabase(config.dbUrl);
startServer(config, db);

Edge Case 3: Error Handling — Unhandled Rejections Kill the Process

If a top-level await throws, the module fails to load. Any module that depends on it also fails. In modern Node.js (16+), an unhandled rejection exits the process with code 1. There's no surrounding try/catch unless you write one: 

// ❌ If config.json is missing, this crashes the process
const config = JSON.parse(
  await readFile('config.json', 'utf8')
);

// ✅ Handle the error explicitly
let config;
try {
  config = JSON.parse(await readFile('config.json', 'utf8'));
} catch (err) {
  if (err.code === 'ENOENT') {
    console.warn('No config.json found, using defaults');
    config = { port: 3000 };
  } else {
    console.error('Fatal: cannot read config', err);
    process.exit(1);
  }
}

Edge Case 4: TypeScript Configuration

If you use TypeScript (and you will in full-stack apps), top-level await requires: 

// tsconfig.json
{
  "compilerOptions": {
    "module": "ES2022",     // or "NodeNext"
    "target": "ES2017",     // minimum for async/await
    "moduleResolution": "bundler" // or "NodeNext"
  }
}

Without these settings, TypeScript emits: "Top-level 'await' expressions are only allowed when the 'module' option is set to 'es2022', 'esnext', 'system', 'node16', or 'nodenext'."

Edge Case 5: Testing Frameworks

Some test runners (older versions of Jest, ts-jest) don't fully support ESM or top-level await. If you use top-level await in a module being tested, the test runner may fail to import it. Solutions:

  • Vitest — full ESM support, handles top-level await natively
  • Jest with ESM — requires --experimental-vm-modules flag and specific config
  • Pattern: Keep top-level await in entry points, not in modules your tests import

Summary: When Top-Level Await Is Appropriate

Use CaseTLA OK?Reason
Entry-point script (server.js, cli.js) ✅ Nothing imports it — no waterfall risk
Quick one-off scripts ✅ No module graph to block
Library modules imported by others ⚠️ Caution Blocks all importers — use async factory instead
Modules with slow network I/O ❌ Avoid Serializes startup, degrades boot time
