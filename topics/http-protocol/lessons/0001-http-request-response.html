HTTP — Lesson 01

The Request-Response Cycle

What fetch() hides from you — and what you need to know to build the other side.

You Already Know HTTP (More Than You Think)

Every time you call fetch('/api/users') in React, you're sending an HTTP request. Every time you check response.status === 200, you're reading an HTTP response. You've been a client for 5 years. Now you're going to be the server.

HTTP (HyperText Transfer Protocol) is a text-based, request-response protocol. That means:

  • Text-based — the messages are human-readable (headers are plain text)
  • Request-response — the client always initiates, the server always responds
  • Stateless — each request is independent (the server doesn't remember previous requests)

Anatomy of an HTTP Request

When your React app calls fetch('https://api.example.com/users?page=2'), this is the raw text that goes over the wire: 

GET /users?page=2 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer eyJhbG...
User-Agent: Mozilla/5.0 ...

Four parts:

PartExamplePurpose
Request line GET /users?page=2 HTTP/1.1 Method + path + protocol version
Headers Host: api.example.com Metadata (key: value pairs)
Blank line (empty) Separates headers from body
Body (optional) {"name":"Mohamed"} Data payload (POST, PUT, PATCH)

Anatomy of an HTTP Response

The server sends back:

HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 127
Cache-Control: max-age=60

[{"id":1,"name":"Mohamed"},{"id":2,"name":"Ahmed"}]
PartExamplePurpose
Status line HTTP/1.1 200 OK Protocol + status code + reason phrase
Headers Content-Type: application/json Metadata about the response
Blank line (empty) Separates headers from body
Body [{"id":1,...}] The actual data

HTTP Methods — The Action Verbs

You use these in fetch() already. Now you'll handle them on the server:

MethodPurposeHas Body?Idempotent?
GETRead/retrieve a resourceNoYes
POSTCreate a new resourceYesNo

DELETERemove a resourceRarelyYes
HEADSame as GET but no body (metadata only)NoYes

Idempotent means: calling it multiple times produces the same result as calling it once. DELETE /users/5 twice → user 5 is deleted (same state). POST /users twice → two users created (different state). This matters for retry logic and caching.

You already know 200 and 404. Here's the full picture a backend dev needs:

Senior-level distinction: 401 means "who are you?" (authentication). 403 means "I know who you are, but you can't do this" (authorization). Getting these right makes your API predictable for frontend developers.

The Connection to Node.js

On the server side, Node.js gives you the raw pieces of this protocol: 

import http from 'node:http';

const server = http.createServer((req, res) => {
  // req = the incoming HTTP request
  //   req.method  → 'GET', 'POST', etc.
  //   req.url     → '/users?page=2'
  //   req.headers → { 'content-type': 'application/json', ... }
  //   req itself  → a Readable stream (the body)

  // res = the outgoing HTTP response
  //   res.statusCode = 200
  //   res.setHeader('Content-Type', 'application/json')
  //   res.end(JSON.stringify(data))
  //   res itself  → a Writable stream
});

server.listen(3000);
Key insight from your fs lessons: req is a Readable stream (like createReadStream) and res is a Writable stream (like createWriteStream). Everything you learned about streams, Buffers, and pipeline() applies directly here. The request body arrives in chunks. The response can be streamed.

HTTP is Stateless — What That Means for You

Each HTTP request is completely independent. The server doesn't remember previous requests from the same client. This is why:

  • Sessions exist — cookies or tokens carry identity across requests
  • Every request must include auth — the server can't "remember" you're logged in
  • REST APIs are stateless — all state lives in the URL, headers, or body

As a frontend dev, you deal with this by attaching tokens to every fetch(). As a backend dev, you'll verify those tokens on every request.
