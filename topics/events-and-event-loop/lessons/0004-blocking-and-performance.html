Events — Lesson 04

Blocking & Performance

What blocks the event loop, how to detect it, and how to fix it without Worker Threads.

What "Blocking" Means in Practice

The event loop can only process one callback at a time. If that callback takes 100ms of CPU time (not I/O wait — actual computation), then for 100ms:

  • No incoming HTTP requests are handled
  • No timer callbacks fire
  • No I/O callbacks run
  • Your entire server is frozen

Detecting Event Loop Lag

// Simple event loop lag detector
let lastCheck = Date.now();

setInterval(() => {
  const now = Date.now();
  const lag = now - lastCheck - 1000; // Expected: 0ms lag
  if (lag > 100) {
    console.warn(`⚠️ Event loop lag: ${lag}ms`);
  }
  lastCheck = now;
}, 1000);

If the interval fires after 1200ms instead of 1000ms, there's 200ms of lag — something blocked the loop for 200ms.

Fix: Chunking with setImmediate

Break a long-running loop into chunks, yielding control back to the event loop between each chunk: 

// ❌ Blocks: processes 1M items without yielding
function processAllSync(items) {
  for (const item of items) {
    heavyComputation(item);
  }
}

// ✅ Non-blocking: processes in chunks, yields between them
function processInChunks(items, chunkSize = 1000) {
  return new Promise((resolve) => {
    let i = 0;

    function processChunk() {
      const end = Math.min(i + chunkSize, items.length);
      for (; i < end; i++) {
        heavyComputation(items[i]);
      }
      if (i < items.length) {
        setImmediate(processChunk); // yield, then continue
      } else {
        resolve();
      }
    }

    processChunk();
  });
}
Why setImmediate and not setTimeout(fn, 0)? setImmediate runs in the check phase — right after I/O. setTimeout(fn, 0) has a minimum ~1ms delay and runs in the timers phase. setImmediate is more predictable for yielding.

Fix: Worker Threads for Heavy CPU Work

// main.js — offload CPU work to a worker
import { Worker } from 'node:worker_threads';

function runInWorker(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker('./heavy-task.js', {
      workerData: data,
    });
    worker.on('message', resolve);
    worker.on('error', reject);
  });
}

// heavy-task.js — runs in a separate thread
import { parentPort, workerData } from 'node:worker_threads';

const result = expensiveComputation(workerData);
parentPort.postMessage(result);

Worker threads run on a separate OS thread — they don't block the main event loop. Use them for: image processing, PDF generation, heavy crypto, large data transforms.

The libuv Thread Pool

Some Node.js built-in operations already use a thread pool (managed by libuv):

  • fs operations (readFile, writeFile, stat) — thread pool
  • crypto.pbkdf2, crypto.randomBytes — thread pool
  • dns.lookup — thread pool
  • Network I/O (TCP, HTTP) — OS kernel async (NOT thread pool)

Default thread pool size: 4 threads. If you have 10 concurrent readFile calls, only 4 run in parallel — the rest queue. Increase with: 

// Set before any I/O — usually at process start
process.env.UV_THREADPOOL_SIZE = '16'; // max: 1024

Summary: Event Loop Rules

  1. Never do CPU-heavy work synchronously in a server request handler
  2. Never use *Sync methods in concurrent code (HTTP servers, workers)
  3. Chunk long loops with setImmediate to yield between iterations
  4. Offload CPU work to Worker Threads for operations > 50ms
  5. Monitor event loop lag in production to catch regressions
